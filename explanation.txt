Travel Itinerary Planner - Selenium Test Code Explanations (Simplified Version)

1) Why did we remove try-catch?

Your updated test files now follow a simple linear script style.

- You asked to avoid the try-catch concept.
- So each test now runs step by step from top to bottom.
- At the end of each test, `driver.quit()` is called directly.

This makes the code easier to read for beginners because:
- Less nesting
- Fewer extra blocks
- Easier to follow execution order

Important note:
- Without try-catch, if any step fails in between, the program stops immediately and browser cleanup may not happen.
- This is acceptable for simple learning scripts, but in production automation teams usually use try-catch/finally or test framework hooks.

2) What libraries were imported and what are they used for?

Below are imports used across Test1-Test4 and their purpose.

Core Selenium classes:
- org.openqa.selenium.WebDriver
  Purpose: Main browser control interface (open pages, navigate, manage window, quit).

- org.openqa.selenium.chrome.ChromeDriver
  Purpose: Chrome-specific driver implementation of WebDriver.

- org.openqa.selenium.By
  Purpose: Locator strategy to find elements (By.id, By.tagName, etc.).

- org.openqa.selenium.WebElement
  Purpose: Represents an HTML element to interact with (sendKeys, click, getText).

- org.openqa.selenium.Dimension
  Purpose: Represents window size (width/height) for setSize/getSize in window management.

Advanced interaction/waiting:
- org.openqa.selenium.support.ui.WebDriverWait
  Purpose: Explicit wait utility until condition is true (for reliable synchronization).

- org.openqa.selenium.support.ui.ExpectedConditions
  Purpose: Prebuilt wait conditions (titleContains, urlContains, visibilityOfElementLocated).

- org.openqa.selenium.support.ui.Select
  Purpose: Handles HTML <select> dropdowns (selectByValue, getFirstSelectedOption).

JavaScript execution:
- org.openqa.selenium.JavascriptExecutor
  Purpose: Runs JavaScript inside browser context (used for localStorage checks and date input setting).

Java standard library:
- java.time.Duration
  Purpose: Defines wait timeout duration (for WebDriverWait).

After simplification:
- Duplicate WebDriverWait imports were removed.
- Unused imports were removed from each test file.

3) Which automation testing concepts are implemented?

Your scripts implement multiple real-world UI automation concepts:

A) Functional UI testing
- Verifies app features like register, login, navigation, and create trip flow.

B) End-to-end workflow testing
- Registers user -> logs in -> reaches dashboard.
- Login prerequisite before navigation/create-trip tests.

C) Browser/window management testing
- maximize, minimize, restore, fullscreen, custom size.

D) Navigation testing
- Direct navigation and browser controls: back, forward, refresh.

E) Synchronization strategy
- Explicit waits via WebDriverWait + ExpectedConditions.
- Static waits via Thread.sleep (used heavily for visual delay and timing).

F) Form validation and data entry automation
- Filling text fields, password fields, date fields, and dropdown values.

G) Dynamic test data generation
- Unique email with System.currentTimeMillis() avoids duplicate-user conflicts.

H) State/session verification
- localStorage check (tripPilotUser) confirms successful login session persistence.

I) Basic reporting/logging
- Console output for title, URL, field values, pass/fail markers.

J) Basic cleanup handling
- driver.quit() is called at the end of each test.

4) Detailed explanation of the 4 code files

A) Test1_WindowManagement
Goal:
- Verify browser window state operations work correctly.

Step-by-step flow:
1. Setup
- Sets ChromeDriver executable path.
- Creates Chrome browser instance.
- Creates explicit wait object (though not used in this test).

2. Open home page
- Navigates to /index.html.
- Waits 3 seconds.

3. Maximize check
- Calls maximize().
- Reads window size with getSize().
- Prints width/height.

4. Minimize and restore
- Calls minimize().
- Waits and logs minimized state.
- Restores using maximize() (this is your required reopen behavior).
- Logs restoration.

5. Fullscreen check
- Calls fullscreen().
- Reads and prints size.

6. Custom resize check
- Sets size to 1280x720.
- Reads and prints final size.

7. Completion
- Prints completion message.
- Quits browser at end.

Testing concept shown strongly:
- Window management APIs and viewport behavior.

B) Test2_RegisterAndLogin
Goal:
- Validate registration flow and then login flow using same generated user.

Step-by-step flow:
1. Setup
- Creates driver, wait, and JavascriptExecutor.

2. Register scenario
- Opens register page and waits for title.
- Prints title, URL, and form visibility.
- Fills name, generated unique email, password, confirm password.
- Submits form.
- Waits for success alert visibility.
- Reads alert text.
- Waits for redirect to login page and confirms URL.

3. Login scenario
- Opens login page and waits for title.
- Locates email/password/button elements.
- Uses the same generated email and password.
- Clicks login.
- Waits for redirect to dashboard.

4. Session verification
- Executes JS to read localStorage key tripPilotUser.
- Prints whether session data exists.

5. Completion
- Prints completion and quits browser at end.

Testing concept shown strongly:
- Multi-step functional + end-to-end user journey with state validation.
